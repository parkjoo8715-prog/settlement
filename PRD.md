# 정산시스템 솔루션 PRD v0.2

## 1. 제품 개요

### 1.1 제품명
- 정산시스템 솔루션

### 1.2 제품 목적
- 하나의 모임 안에서 여러 차수의 결제 내역과 참석 인원을 관리하고, 참석자별 최종 정산 금액을 자동으로 계산하는 웹 솔루션을 제공한다.
- 벙주 할인 규칙, 신입 선입금 규칙, 차수별 참석 인원 변동을 자동 반영하여 수기 계산 오류를 제거한다.

### 1.3 핵심 가치
- 차수별 참석 인원이 달라도 자동 배분된다.
- 벙주 할인 규칙이 차수별로 자동 적용된다.
- 신입 선입금 3만원을 반영한 최종 추가 입금액 또는 환불액이 자동 계산된다.
- 차수별 데이터와 전체 총합 데이터를 동시에 제공한다.

## 2. 문제 정의

- 기존 정산은 차수별 참석 인원이 달라질 때 수동 계산이 복잡하다.
- 벙주 할인 규칙이 차수와 인원에 따라 달라져 실수가 발생하기 쉽다.
- 신입 선입금 3만원을 포함한 최종 차액 계산이 번거롭다.
- 여러 차수를 진행한 뒤 전체 합산 결과를 한 번에 보기 어렵다.

## 5. 도메인 구조

### 5.1 모임
- 하나의 정산 단위다.
- 동일 모임 안에 여러 차수를 생성할 수 있다.
- 벙주, 입금 계좌, 참석자 마스터 목록을 가진다.

### 5.2 차수
- 1차, 2차, 3차처럼 순차적으로 증가한다.
- 차수 번호는 1 이상이며 제한 없이 계속 추가할 수 있다.
- 차수마다 결제 총금액과 실제 참석자 목록이 다를 수 있다.

### 5.3 참석자 마스터
- 닉네임 기준으로 모임 전체 참석자 후보를 관리한다.
- 벙주 여부, 신입 여부를 가진다.
- 벙주는 1명만 존재하며 신입일 수 없다.

### 5.4 차수 참석자
- 각 차수에서 실제 참석한 사람의 목록이다.
- 차수별 참석자 수에는 벙주가 포함된다.

## 7. 비즈니스 규칙

### 7.1 차수 규칙
- 차수는 1차부터 시작하며 4차, 5차, 6차처럼 계속 증가할 수 있다.
- 각 차수는 독립적으로 결제 총금액과 참석자를 가진다.

### 7.2 참석 인원 규칙
- 참석자 수는 벙주를 포함한 실제 참석 인원이다.
- 차수마다 참석 인원이 달라도 된다.

### 7.3 벙주 규칙
- 벙주는 모임당 1명이다.
- 벙주는 모든 차수에서 참석자로 간주한다.
- 벙주는 신입으로 등록할 수 없다.

### 7.4 신입 규칙
- 신입은 모임 기준 선입금 30,000원을 가진다.
- 신입의 최종 차액은 모임 전체 누적 부담 금액을 기준으로 계산한다.
- 신입 최종 차액이 0보다 크면 추가 입금 대상이다.
- 신입 최종 차액이 0보다 작으면 환불 대상이다.
- 신입 최종 차액이 0이면 정산 완료다.

### 7.5 벙주 할인 규칙
- 참석 인원이 6명 미만이면 벙주 할인 금액은 0원이다.
- 1차 또는 2차에서 참석 인원이 6명 이상이면 벙주 할인 금액은 10,000원이다.
- 3차 이상에서 참석 인원이 8명 미만이면 벙주 할인 금액은 0원이다.
- 3차 이상에서 참석 인원이 8명 이상이면 벙주 할인 금액은 20,000원이다.

### 7.6 자동 배분 규칙
- 사용자는 `인당 받을 금액`을 직접 입력하지 않는다.
- 시스템은 차수별 결제 총금액과 참석 인원을 기준으로 참석자별 부담 금액을 자동 계산한다.
- 벙주 할인 금액은 해당 차수의 나머지 참석자들에게 자동 배분된다.
- 자동 배분 결과 각 차수의 참석자별 부담 금액 합계는 해당 차수 결제 총금액과 정확히 같아야 한다.

### 7.7 총합 규칙
- 전체 총합 데이터는 모든 차수의 결과를 합산해 계산한다.
- 참석자별 최종 정산 금액은 자신이 참석한 차수들의 부담 금액 합계다.
- 신입은 최종 정산 금액에서 선입금 30,000원을 차감하여 최종 차액을 계산한다.

## 8. 계산식 정의

### 8.1 기본 변수
- `roundTotal = 차수별 결제 총금액`
- `participantCount = 해당 차수 참석 인원 수`
- `roundNumber = 차수 번호`
- `hostDiscount = 벙주 할인 금액`
- `newMemberAdvance = 30000`

### 8.2 벙주 할인 계산
- `participantCount < 6` 이면 `hostDiscount = 0`
- `participantCount >= 6` 이고 `roundNumber < 3` 이면 `hostDiscount = 10000`
- `roundNumber >= 3` 이고 `participantCount < 8` 이면 `hostDiscount = 0`
- `participantCount >= 8` 이고 `roundNumber >= 3` 이면 `hostDiscount = 20000`

### 8.3 차수별 자동 배분 계산
- `rawHostAmount = max((roundTotal / participantCount) - hostDiscount, 0)`
- `nonHostCount = participantCount - 1`
- `rawNonHostAmount = nonHostCount > 0 ? (roundTotal - rawHostAmount) / nonHostCount : roundTotal`
- 벙주 부담 금액은 `rawHostAmount`를 기준으로 계산한다.
- 일반 참석자 부담 금액은 `rawNonHostAmount`를 기준으로 계산한다.
- 원 단위 정산을 위해 소수점은 버림 처리한다.
- 버림 처리 후 남는 잔여 금액은 해당 차수의 비벙주 참석자에게 1원씩 순차 배분한다.
- 비벙주 참석자가 없는 경우 잔여 금액은 벙주에게 배분한다.

### 8.4 참석자별 누적 금액 계산
- `participantTotal = 자신이 참석한 모든 차수 부담 금액의 합`

### 8.5 신입 최종 차액 계산
- 신입 최종 차액 = `participantTotal - newMemberAdvance`
- `최종 차액 > 0`: 추가 입금
- `최종 차액 < 0`: 환불
- `최종 차액 = 0`: 정산 완료

### 8.6 전체 총합 계산
- `grandTotalPayment = 모든 차수 결제 총금액의 합`
- `grandTotalAssigned = 모든 참석자 누적 부담 금액의 합`
- `grandTotalAdvance = 신입 선입금 총합`
- `grandTotalAdditional = 추가 입금 대상 금액 합`
- `grandTotalRefund = 환불 대상 금액 합`
- `grandTotalPayment` 와 `grandTotalAssigned` 는 동일해야 한다.

## 9. 기능 요구사항

### 9.1 홈페이지
- 서비스명, 문제 해결 포인트, 핵심 기능이 보여야 한다.
- 사용자는 홈페이지에서 바로 정산 시작 화면으로 이동할 수 있어야 한다.
- 모바일과 PC에서 모두 사용 가능해야 한다.

### 9.2 모임 생성 기능
- 벙주 닉네임을 입력할 수 있어야 한다.
- 모임명을 입력할 수 있어야 한다.
- 입금 계좌를 입력할 수 있어야 한다.
- 참석자 마스터 목록을 추가, 수정, 삭제할 수 있어야 한다.
- 벙주는 정확히 1명만 지정할 수 있어야 한다.
- 벙주를 신입으로 지정할 수 없어야 한다.

### 9.3 차수 관리 기능
- 사용자는 차수를 여러 개 추가할 수 있어야 한다.
- 각 차수에 대해 결제 총금액을 입력할 수 있어야 한다.
- 각 차수에 대해 실제 참석자를 선택할 수 있어야 한다.
- 차수 번호는 자동 증가해야 한다.
- 특정 차수는 수정 또는 삭제할 수 있어야 한다.

### 9.4 자동 계산 기능
- 차수별 결제 총금액 입력 시 즉시 자동 배분 결과가 계산되어야 한다.
- 참석자 추가 또는 제거 시 벙주 할인과 참석자별 부담 금액이 즉시 재계산되어야 한다.
- 차수 번호가 3 이상이 되는 경우 8인 이상 조건의 추가 할인 여부가 즉시 반영되어야 한다.
- 전체 총합 데이터는 차수 변경 시 즉시 업데이트되어야 한다.
- 신입 최종 차액은 전체 누적 금액 기준으로 자동 계산되어야 한다.

### 9.5 차수별 결과 화면
- 차수 번호를 표시해야 한다.
- 차수별 결제 총금액을 표시해야 한다.
- 참석 인원 수를 표시해야 한다.
- 벙주 할인 금액을 표시해야 한다.
- 차수별 참석자 목록과 각자의 부담 금액을 표시해야 한다.
- 차수별 합계가 결제 총금액과 일치함을 보여줘야 한다.

### 9.6 전체 총합 결과 화면
- 전체 차수 수를 표시해야 한다.
- 전체 결제 총금액 합계를 표시해야 한다.
- 전체 신입 선입금 총액을 표시해야 한다.
- 전체 추가 입금 총액을 표시해야 한다.
- 전체 환불 총액을 표시해야 한다.
- 참석자별 누적 부담 금액을 표시해야 한다.
- 신입의 경우 선입금 반영 후 최종 차액 상태를 표시해야 한다.
- 벙주를 포함한 모든 참석자의 최종 정산 상태를 한 번에 확인할 수 있어야 한다.

### 9.7 공유 기능
- 차수별 결과를 텍스트로 복사할 수 있어야 한다.
- 전체 총합 결과를 텍스트로 복사할 수 있어야 한다.
- 모바일 메신저 공유에 적합한 형태로 요약 문구를 제공해야 한다.

## 11. UX 원칙

- 한 화면에서 흐름이 끊기지 않도록 입력과 결과를 바로 연결한다.
- 차수별 데이터와 최종 총합 데이터가 시각적으로 명확히 구분되어야 한다.
- 계산 규칙은 툴팁 또는 안내 문구로 항상 확인할 수 있어야 한다.
- 모바일에서 차수 추가와 참석자 선택이 쉬워야 한다.
- 입력 변경 시 새로고침 없이 즉시 결과가 반영되어야 한다.

## 12. 검증 조건

- 벙주는 반드시 1명이어야 한다.
- 벙주는 신입일 수 없다.
- 차수별 결제 총금액은 0보다 커야 한다.
- 차수별 참석자는 최소 1명 이상이어야 한다.
- 차수별 참석자 목록에는 반드시 벙주가 포함되어야 한다.
- 참석자 닉네임은 비어 있으면 안 된다.
- 같은 모임 안에서 참석자 닉네임은 중복되면 안 된다.
- 차수 번호는 1 이상의 정수여야 한다.
- 차수별 배분 합계는 반드시 차수별 결제 총금액과 일치해야 한다.

## 13. 성공 지표

- 벙주가 여러 차수 정산을 3분 이내에 완료할 수 있다.
- 차수별 참석 인원 변동이 있어도 추가 수기 계산 없이 정산할 수 있다.
- 신입 선입금 반영 결과를 사용자가 별도 계산 없이 바로 이해할 수 있다.
- 모바일에서도 결과 복사와 공유까지 완료할 수 있다.

## 14. 현재 정책 가정

- 신입 선입금 30,000원은 `모임 전체 기준 1회 선입금`으로 처리한다.
- 따라서 신입의 최종 차액은 `자신이 참석한 모든 차수의 누적 부담 금액 - 30,000원`으로 계산한다.
- 이 가정은 차수별 데이터와 최종 총합 데이터를 함께 보여주려는 현재 요구사항에 가장 적합하다.
